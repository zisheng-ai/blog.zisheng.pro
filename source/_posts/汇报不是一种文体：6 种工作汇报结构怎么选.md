*先判断读者需要同步状态、解决问题、作出决定，还是校准经营动作，再选汇报结构。*

“一句话结论、完成项、风险、遗留项、Action Required”这套写法很实用。我把它理解为一种高密度的项目状态汇报：领导先知道现在是否可控，再决定要不要介入。它的价值不在于“阿里风格”四个字，而在于把事实、风险和责任压进同一个阅读顺序。

但我也踩过反面例子：方案还没想清楚，就硬写“已完成”；一个需要拍板的问题，被塞进周报最后一行；复盘写成“下次加强”。格式没错，沟通却没有发生。

## 一句话总结

汇报是一种面向特定读者的决策接口。先问读者读完后应该做什么：**知道状态、排除问题、批准选择、投入资源、校准经营，还是修正机制**。答案不同，结构也应该不同。

我把最常见的六类放进一张表：

| 场景 | 读者真正的问题 | 推荐结构 | 不适合的情况 |
| --- | --- | --- | --- |
| 状态汇报 | 事情是否还可控？ | 结论 → 完成项 → 风险 → 下步 → Action Required | 需要讨论方案细节 |
| 问题汇报 | 出了什么事，如何止损？ | 事实 → 根因 → 影响 → 处置方案 → 需要支持 | 只是日常进度波动 |
| 决策汇报 | 现在该选哪条路？ | 建议 → 选项 → 取舍 → 决策人 → 截止时间 | 选项尚未形成 |
| 方案汇报 | 值不值得投入？ | 目标 → 路径 → 证据 → 成本 → 风险 | 只想同步既有进展 |
| 经营复盘 | 指标为何变化，下一周期怎么调？ | 结果 → 趋势 → 归因 → 经营动作 → 验证指标 | 单点事故排障 |
| 复盘汇报 | 哪个机制失效，如何避免再犯？ | 预期 → 经过 → 偏差 → 根因 → 机制改动 | 需要立即止损的事故 |

## 1. 状态汇报：把“我做了什么”翻译成“是否需要你介入”

你给出的模板属于这一类：

```text
【一句话结论】当前状态与判断
【完成项】结果、数字、发布状态
【风险/问题】按严重度说明影响和处理
【遗留项】未完成事项、原因、预计时机
【Action Required】需要谁在何时决定或提供什么
```

它的核心不是完整，而是**让读者用十秒判断项目健康度**。项目状态报告的通用做法也是先给高层摘要，再明确里程碑、阻塞、风险、指标和下一步；风险必须被写出来，才有机会被提前处理。[Atlassian 的项目状态报告指南](https://www.atlassian.com/agile/project-management/status-report)把这几个要素列得很完整。

我会给状态汇报加一条硬约束：每一个风险都要写出 `影响 + Owner + 触发条件 + 当前动作`。没有 Owner 的风险只是焦虑；没有触发条件的风险无法判断优先级。

```text
Warning：发布依赖的权限审批尚未完成。
影响：若周四 18:00 前未完成，周五灰度窗口将顺延一周。
Owner：张三。
当前动作：今天完成审批材料，明天上午同步审批结果。
```

## 2. 问题汇报：不要把事故写成进度更新

问题汇报要先让读者知道事实边界，不能只写“我们正在处理”。最短也要回答五件事：

1. **事实**：发生了什么，何时发现，是否已经止住。
2. **根因**：已经确认的根因与仍待验证的假设分开写。
3. **影响**：用户、收入、交付、数据或安全分别受什么影响。
4. **方案**：临时止损和长期修复不能混为一谈。
5. **需要支持**：谁需要在什么时候提供什么资源或决策。

```text
【问题】iOS 登录回调在 14:05–14:28 出现失败。
【影响】新用户登录失败率 18%，存量登录不受影响；14:31 已回滚。
【根因】回调参数兼容逻辑遗漏了一个空值分支，已复现。
【处置】今晚补齐回归用例，明日灰度；长期将该类参数校验前置到网关。
【需要支持】无。
```

这里最重要的纪律是：不确定就写“不确定”。把推测写成根因，会让后续排查被错误方向锁死。

## 3. 决策汇报：目标是让人拍板

当你真正需要的是一个选择，就不要发一篇泛泛的方案说明。决策文档应该让决策人看完后能回答：**选什么、不选什么、代价是什么、谁来落地、最晚何时决定**。

Atlassian 的 [Decision 模板](https://www.atlassian.com/dam/jcr%3Ac13e0440-f990-4e13-b226-1f3cf6eb37d4/Decision.pdf)把决策状态、影响、驱动者、批准者、参与者和截止时间拆开，重点正是避免“大家都讨论过，但没有人负责决定”。

我的最小模板是：

```text
【建议】选方案 B：先接现成 Provider，再补 Adapter 层。
【原因】交付时间最短；未来替换成本可控。
【备选】A 更完整，但首期多 3 周；C 最便宜，但会锁死协议。
【代价】首期少做两项高级能力，第二阶段补齐。
【需要决定】请产品负责人在周三 17:00 前确认 B；确认后由我在本周启动实现。
```

“推荐方案”必须写在最前面。让读者在三个选项中自行猜你的立场，等于把汇报对象变成分析师。

## 4. 方案汇报：用证据说明为什么值得投入

方案汇报面向的是“要不要做、做多大”的问题。它和决策汇报相近，但重点不同：决策汇报强调选项和拍板，方案汇报强调目标、路径与资源合理性。

Amazon 的 Narrative 方法要求把规划写成连续论证。AWS 的公开说明提到，这类叙事文档用于团队和组织的年度规划，正文上限通常为六页，附件另算。[AWS Startups 对 Narrative 的介绍](https://aws.amazon.com/blogs/startups/startup-advice-how-to-write-a-narrative/)值得参考，但不必机械照搬页数。

我会用这五段：

```text
【目标】要解决谁的什么问题，成功标准是什么。
【现状】现有流程、数据或用户证据表明什么。
【路径】做什么、不做什么，按什么阶段验证。
【投入】人力、时间、依赖与机会成本。
【风险】最大的失败路径，以及何时止损或调整。
```

方案最容易失真的地方是“愿景写得很大，验证点写得很少”。只要没有量化成功标准和停止条件，它更像倡议，不像可执行方案。

## 5. 经营复盘：解释数字会怎么改变动作

经营复盘的单位不是任务，而是一个可持续观察的业务结果：留存、转化、履约时效、成本、收入、质量或效率。它最忌讳只报“同比增长 20%”，因为读者下一秒会问：为什么、能否持续、接下来怎么做？

一个完整的经营复盘至少要把结果和动作接起来：

```text
【结果】本月激活率 31%，较上月 +4.2pp，目标 34%。
【趋势】增长主要发生在新渠道，老渠道持平。
【归因】新手引导缩短带来首日完成率提升；尚不能证明是唯一原因。
【动作】下月把引导实验扩到老渠道，同时保留对照组。
【验证】看首日完成率、7 日留存与投诉率，任一质量指标恶化即回滚。
```

这里的关键是把“归因”与“相关”分开。没有实验、对照或足够证据时，写“可能原因”比写“根因”更专业。

## 6. 复盘汇报：把“下次注意”改成可验证的机制

复盘要落到机制改动。它应该回答：原本期待什么、实际上发生了什么、差异在哪、为什么现有机制没有提前发现、下次怎样让系统自己更难犯同类错误。

| 层级 | 无效写法 | 有效写法 |
| --- | --- | --- |
| 人 | 加强重视 | 发布前增加双人校验，Owner 明确到角色 |
| 流程 | 优化流程 | 将权限检查移到发布流水线的必经关卡 |
| 工具 | 完善监控 | 对关键错误率设阈值，连续 5 分钟超标自动告警 |
| 指标 | 持续关注 | 下周复盘告警命中率与误报率 |

复盘的交付物应落到机制变更：一条规则、一个测试、一项监控、一个检查点，或者明确删掉一项无效流程。

## 不属于汇报的两类沟通

有两类内容经常被塞进汇报，结果两边都做不好。

- **反馈**不是汇报。对人的行为反馈，更适合用 `Situation → Behavior → Impact`：说清场景、可观察行为和造成的影响，避免把评价包装成结论。[Center for Creative Leadership 的 SBI 工具说明](https://www.facs.org/media/pshbyz4v/sbi-feedback.pdf)强调的也是具体情境与行为，而不是泛泛评价。
- **求助**也不是汇报。求助信应该先写“我卡在哪里、已经试过什么、希望你提供哪一种帮助”，不需要补一整段项目历史。

## 我的判断框架：先选读者的下一步动作

我现在不再先问“该写周报还是写 PPT”，而是先给读者分配下一步动作：

```text
只需要知道是否可控        → 状态汇报
需要排除一个正在发生的问题 → 问题汇报
需要在选项之间作出承诺     → 决策汇报
需要批准投入与方向         → 方案汇报
需要调整目标和经营动作     → 经营复盘
需要修正系统性失误         → 复盘汇报
```

“阿里风格”的价值，在于它要求写作者先给结论、再用数字和风险支撑判断、最后把责任落到行动上。这个顺序应该保留；外层模板则随问题改变。

高质量汇报要让对方读完后知道：现在发生了什么、我该不该介入、下一步由谁在什么时候做什么。
