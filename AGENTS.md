# 博客项目约定

## 封面图决策

封面不是默认的氛围图；先判断文章在列表页最需要传递什么。

- **文字 KV / 方案封面**：适用于选型、架构方案、项目复盘、工具/产品发布、路线图、对比和方法论。封面必须在一眼内给出标题或核心结论，并用 2–4 个模块、对比项或流程节点表达信息结构。GBrain、Sirchmunk、Understanding Anything 的三层搭配属于这一类；视觉上可采用黑客松作品封面或产品 KV 的表达，但信息必须服务文章结论。
- **纯氛围图**：适用于技术随笔、排障复盘、体验记录、观点文章和叙事性内容。它负责建立情绪与技术气质，不应伪装成信息图，也不强行添加文字。
- **混合型封面**：适用于教程与工具实战。在氛围背景上保留一个短标题、流程关键词或关键对象，既保持阅读气质，也让读者预判主题。
- **架构封面必须进入正文**：文章使用架构图或架构型文字 KV 作为封面时，同一张架构大图必须作为正文第一张图片，放在 front matter 之后、引言之前；不能只在页面头图或列表卡片中展示。
- **架构图不生成氛围底图**：架构图优先用 HTML/CSS、SVG、Mermaid 或 draw.io 确定性绘制，背景使用纯色或 CSS 渐变。不要调用图像模型生成氛围背景再合成；只有用户明确要求插画化或氛围化表达时才例外。

判断顺序：先问“读者在列表页是否需要立即获得结构化结论”；需要则选文字 KV，不需要则优先氛围图。禁止因统一视觉而把所有文章做成同一种封面。

## 公开脱敏

写入公开博客前，检查标题、description、正文、代码块、标签、图片/SVG 文字和资源标识。项目代号、内部产品名、内部域名、仓库路径与具体技术栈若会暴露未公开实现，统一替换为不指向具体项目的表述，例如“应用侧”“原生 Core”“本地 BM25 索引”。不在公开文章中保留项目代号或可反向定位的内部资源名称。

## Coding Agent 汇报

汇报对象是能继续执行、验证或审查代码的技术负责人，不是泛读进度的管理者。优先交付可核验状态，少写过程叙述和泛泛结论。

```text
【结论】目标是否完成；若未完成，明确卡点。
【变更】N 个文件 / N 项行为变化；列出关键路径和有意未动的范围。
【验证】实际执行的命令、测试或视觉检查及结果；未验证要写原因。
【状态】分别写工作区、暂存、提交、推送、部署的真实状态，禁止把其中任一项推断成另一项。
【风险/阻塞】Critical / Warning / Info；每项写影响、Owner 和下一步。没有就写“无”。
【Action Required】仅列出需要用户授权、外部输入或明确决策的事项；没有就省略本段。
```

- 不用“已完成”“已修复”“已发布”这类无证据状态词；后面必须跟文件、命令、commit、URL 或可复现现象。
- 报告“改了什么”，不逐条复述“我做了什么”。只列与目标相关的文件、接口、数据或行为。
- 测试通过只说明已验证的范围；构建通过不等于线上已部署，推送成功不等于部署已验证。
- 遇到失败，先给根因、影响和下一步，再给日志摘要；不贴大段原始日志。
- 不要求用户重复确认已在当前任务范围内的普通实现步骤；只有权限、外部系统或范围扩大需要确认时才列入 Action Required。
